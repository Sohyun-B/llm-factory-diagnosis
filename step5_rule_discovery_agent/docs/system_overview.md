# 자연어 규칙 발견 시스템: 설계와 차별점

> 작성일: 2026-03-10
> 기반: step5_rule_discovery_agent 실험 결과 (Exp 1~5, 합성 데이터 10개 패턴)

---

## 1. 이 시스템은 무엇인가

B2B 자재 발주 이력에서 **예측에 쓸 수 있는 반복 패턴을 자연어 규칙으로 자동 발견**하는 시스템이다.

```
입력: 발주 이력 CSV (누가, 언제, 뭘, 얼마나)
출력: "한진산업은 매월 첫째 주에 STS304를 약 100kg 주문한다" (confidence=0.95)
```

핵심은 ML 모델의 가중치가 아니라 **사람이 읽고 판단할 수 있는 문장**이 예측의 근거라는 것이다.

---

## 2. 작동 방식: 3단계 파이프라인

```
Phase 1                    Phase 2                     Phase 3
통계 도구 (자동)            LLM (추론)                   검증 도구 (자동)
┌──────────────┐          ┌──────────────┐            ┌──────────────┐
│ 8-Layer 분석  │  ──→     │ 패턴 해석     │   ──→      │ 정답 대조     │
│ - 간격/주기   │  통계    │ - "왜?" 추론  │   규칙     │ - 매칭 검증   │
│ - GMM 분해   │  요약    │ - 규칙 생성   │   JSON     │ - 거짓양성    │
│ - drift 감지  │         │ - 도구로 조회  │            │ - 보고서 생성  │
└──────────────┘          └──────────────┘            └──────────────┘
phase1_generic.py          LLM 서브에이전트             phase3_verify.py
     ↓                          ↓                          ↓
level0_results.json        llm_rules.json           verification_report.md
     ↓
phase2_prompt_builder.py
     ↓
phase2_prompt.md (7.9KB, 핵심 신호만)
```

### Phase 1: 통계 도구가 큰 그림을 잡는다

`phase1_generic.py`가 도메인 지식 없이 8개 레이어의 분석을 자동 수행한다.

| Layer | 분석 | 발견하는 것 |
|-------|------|-----------|
| 0 | 스키마, 분포 | 기본 구조 |
| 1 | 엔티티별 프로파일 | 발주 간격, 요일/주차 집중 |
| **1.5** | **GMM 혼합 분포 분해** | **같은 시계열에 섞인 복수 패턴 분리** |
| 2 | 관계 | 동반 발주, 고객 간 근접 발주 |
| **2.5** | **Component 재분석** | **패턴 분리 후 drift 감지, 선행자 식별** |
| 3 | 시간 구조 | 월별 추세, 계절성 |
| **3.5** | **다중 조건 연관** | **A AND B이면 C (단독 약한 조건의 교차)** |
| 4 | 이벤트→행동 시차 | 견적요청→발주 전환율, 시차 |

Layer 1.5, 2.5, 3.5(★ 표시)이 이 시스템의 핵심이다. 이유는 뒤에서 설명한다.

### Phase 2: LLM이 통계 위에서 "왜?"를 추론한다

`phase2_prompt_builder.py`가 63KB의 분석 결과에서 **핵심 신호만 추출**(7.9KB, 88% 압축)하여 LLM에게 전달한다. 추출 기준:

- 간격 CV < 0.5인 규칙적 패턴 + CV ≥ 0.5인 불규칙 패턴
- GMM 2+ component인 것만
- 고객 간 proximity lift > 1.3인 것만
- cleaned_drift detected인 것만
- 다중 조건 상위 10개
- 이벤트 follow_rate > 50%인 것만

LLM은 이 요약을 보고 규칙을 생성하되, **더 자세히 보고 싶으면 원본 CSV를 직접 조회**할 수 있다.

### Phase 3: 자동 검증

`phase3_verify.py`가 LLM이 발견한 규칙을 심어둔 정답 패턴(P1~P10)과 대조하여 발견율, 거짓양성, 매칭 상세를 보고서로 생성한다.

## 3. 실험 결과: 합성 데이터 10개 패턴

175건의 발주 데이터에 10개 패턴(P1~P10)을 심어두고 발견 여부를 측정했다.

### 심어둔 패턴

| ID | 유형 | 내용 | 난이도 |
|----|------|------|--------|
| P1 | 주기 | 한진산업 STS304, 매월 ~28일 주기 | Easy |
| P3 | 주기 | 대성기업 AL6061, 격주 ~14일 | Easy |
| P6 | 주기 | 세진테크 분기(91일) 3품목 묶음 | Easy |
| P2 | 계절 | 봄철 수량 30% 증가 | Medium |
| P4 | 교차 | 한진산업 발주 후 2~4일 내 대성기업 추가 | Medium |
| P7 | 동반 | STS304 + CU_PIPE 동반 발주 | Medium |
| P8 | 이벤트 | 동아전자 견적→발주 (~10일, 품목일치) | Medium |
| P5 | 예외 | 공휴일 월에 한진산업 W1→W2 밀림 | Hard |
| P9 | 조건부 | 한진산업 대형(150kg+) → 세진테크 STS316 | Hard |
| P10 | drift | 대성기업 AL6061 간격 14일→21일 증가 | Hard |

### 결과 비교

| 실험 조건 | Easy | Medium | Hard | 계 |
|-----------|------|--------|------|----|
| Exp 1: v1 통계만 (Phase 1만) | 3/3 | 2/4 | 1/3 | **6/10** |
| Exp 1: v1 통계 + raw 전체 + LLM | 3/3 | 4/4 | 3/3 | **10/10** |
| **Exp 5: v2 통계(8-Layer) + LLM** | **3/3** | **4/4** | **2/3** | **9/10** |

**v2 통계만으로 9/10.** raw 데이터 전체를 보지 않고도 v1+raw(6/10)보다 3개 더 발견했다.

유일한 미감지 P5(공휴일 예외)는 통계에 공휴일 캘린더 정보가 없어 구조적으로 감지 불가능한 경우다.

## 4. 기존 접근과의 비교

### 4-1. "LLM에게 데이터 던지고 패턴 찾아줘" vs 이 시스템

가장 단순한 접근: LLM에게 raw 데이터를 보여주고 "패턴 찾아줘"라고 하는 것.

```
단순 접근:        LLM ← 데이터 전체
이 시스템:  도구 → 통계 요약 → LLM → 규칙 + 필요시 raw 조회
```

| 비교 항목 | "데이터 던지기" | 이 시스템 |
|-----------|---------------|----------|
| 컨텍스트 | 전체 데이터 (수천~수만행) | 8KB 요약 + 선택적 조회 |
| 발견 품질 | 당연한 소리만 함 ("A가 가장 많이 팔린다") | 교차 패턴, drift, 조건부 트리거까지 발견 |
| 확장성 | 수천 행 넘으면 불가능 | 통계 압축으로 데이터 크기 무관 |
| 재현성 | LLM 응답이 매번 다름 | Phase 1 통계는 결정적, Phase 2만 확률적 |

**핵심 차이**: LLM은 "어디를 볼지" 판단하는 역할이고, "무엇이 있는지"는 통계 도구가 먼저 잡는다.

### 4-2. 기존 통계 분석 도구 vs 이 시스템

DataPrep.EDA, tsfresh, catch22 같은 자동 EDA 도구는 피처를 추출하지만 **해석은 사람이 한다**.

```
기존 도구:   데이터 → 통계 피처 → 사람이 해석
이 시스템:   데이터 → 통계 피처 → LLM이 해석 → 자연어 규칙
```

| 비교 항목 | DataPrep.EDA / tsfresh | 이 시스템 |
|-----------|----------------------|----------|
| 출력 | 피처 테이블, 차트 | 자연어 규칙 ("A는 매월 B를 주문한다") |
| 해석 | 사람 | LLM |
| 혼합 패턴 | 미지원 | GMM 분해 → 패턴 분리 후 개별 분석 |
| 교차 패턴 | 미지원 | 고객 간 proximity, 선행자 식별 |
| 행동 지침 | 없음 | "STS304 100kg 준비 필수" |

### 4-3. 가설 발견 연구 (HypoGeniC, HypoAgents) vs 이 시스템

LLM 기반 가설 생성 연구(HypoGeniC의 Peirce 프레임워크, HypoAgents의 베이지안 확률)는 **가설 관리 프레임워크**를 제시한다.

| 비교 항목 | HypoGeniC / HypoAgents | 이 시스템 |
|-----------|------------------------|----------|
| 가설 생성 | LLM이 직접 데이터에서 가설 생성 | 통계 도구가 단서 추출 → LLM이 해석 |
| 가설 검증 | 새 데이터에 적용 | 규칙별 검증 방법 자동 정의 + 정확도 이력 추적 |
| 가설 폐기 | 통계적 기각 | **생애주기 관리** (경고→퇴역→아카이브→부활) |
| 혼합 패턴 | 미고려 | GMM + cleaned drift로 분해 후 분석 |
| 결과물 | 가설 리스트 | **규칙 카드** (자연어 확신도 + 추세 + 행동 지침) |

**핵심 차이**: 가설 발견 연구는 "발견"에 집중하고, 이 시스템은 **발견 → 검증 → 퇴역 → 부활의 전체 생애주기**를 다룬다.

### 4-4. BRMS (Drools 등) vs 이 시스템

산업용 비즈니스 규칙 관리 시스템(BRMS)은 규칙의 버전 관리, 감사 추적, 런타임 추가/삭제를 지원한다.

```
BRMS:      사람이 규칙 작성 → 시스템이 실행/관리
이 시스템:  도구+LLM이 규칙 발견 → 시스템이 검증/관리
```

| 비교 항목 | BRMS (Drools) | 이 시스템 |
|-----------|--------------|----------|
| 규칙 생성 | 사람이 수동 작성 | LLM이 데이터에서 자동 발견 |
| 규칙 검증 | 사람이 주기적 리뷰 | 자동 검증 + 정확도 이력 |
| Concept Drift | 미지원 | ADWIN / 연속 미스 / 추세 감지 |
| 계절성 | 수동 태깅 | 아카이브 보존 + 자동 부활 |
| 규칙 포맷 | IF-THEN 코드 | 자연어 문장 |

**핵심 차이**: BRMS는 "사람이 아는 규칙을 관리"하고, 이 시스템은 "사람이 모르는 규칙을 데이터에서 발견"한다.

### 4-5. 시계열 예측 모델 (Prophet, ARIMA, LSTM 등) vs 이 시스템

전통적 시계열 예측은 과거 패턴에서 미래 값을 예측하지만, 예측 근거를 설명하지 못한다.

| 비교 항목 | Prophet / LSTM | 이 시스템 |
|-----------|---------------|----------|
| 출력 | 예측값 (수치) | 규칙 (자연어) → 예측 근거 |
| 설명 가능성 | 블랙박스 (특히 LSTM) | "이 예측은 규칙 #001, #007 때문" |
| 다중 원인 | 단일 시계열 모델링 | 교차 패턴, 이벤트 트리거, 조건부 등 다중 원인 |
| Concept Drift | 재학습 필요 | 규칙별 개별 관리 (살리기/죽이기) |
| 사용자 지식 통합 | 불가 | 사용자가 규칙 직접 입력 가능 |

**핵심 차이**: 시계열 모델은 "무엇이 올지" 예측하고, 이 시스템은 **"왜 그것이 올지"를 설명하면서 예측**한다.

## 5. 이 시스템만의 차별점 5가지

### 차별점 1: 혼합 패턴 분해 (Layer 1.5 + 2.5)

**현실 데이터의 핵심 문제**: 같은 고객의 같은 품목이라도 **다른 이유**로 발주한다.

```
대성기업 AL6061 발주 30건:
  = 격주 정기 발주 23건 (P3)
  + 한진산업에 의한 교차 발주 6건 (P4)
  + noise 1건

간격: [15, 14, 13, 14, 14, 1, 14, 13, 0, ...]
                          ↑P4       ↑P4
```

이 혼합 데이터를 그대로 분석하면 drift(14일→21일)가 교차 발주 노이즈에 묻힌다.

**해결**: GMM으로 2개 분포(14.2일 주 패턴 + 2.8일 부 패턴)를 자동 분리 → 부 패턴의 선행자(한진산업/STS304)를 식별 → 교차 발주를 제거한 순수 시계열에서 drift 감지 (Spearman ρ=0.505, p=0.014).

**기존 도구와의 차이**: tsfresh, catch22는 단일 시계열을 가정한다. "같은 시계열 안에 섞인 다른 원인의 데이터"를 분리하는 레이어가 없다.

### 차별점 2: 통계 도구 → LLM 분업 구조

```
잘못된 분업:  LLM이 전부 한다  → 당연한 소리만 함
잘못된 분업:  도구가 전부 한다  → 피처 테이블에서 해석 불가

올바른 분업:
  도구 → "한진산업 STS304 간격 CV=0.20, 12회, 월요일 집중 26%"
  LLM  → "한진산업은 매월 첫째 주 월요일에 STS304를 정기 발주한다.
          공급망 관리 목적의 월초 구매 승인 프로세스에 의한 것으로 추정."
```

LLM은 귀납 추론에 약하다(ACL 2025). 노이즈 데이터에서 정확한 수치 규칙을 직접 추출하지 못한다. 그러나 "CV=0.20이 무엇을 의미하는지", "왜 이런 패턴이 존재하는지" 해석하는 데는 탁월하다.

**실험 증거**: v1 통계만 준 경우 6/10 발견. v2 8-Layer 통계를 준 경우 **9/10 발견**. 통계의 품질이 LLM 발견 품질을 직접 결정한다.

### 차별점 3: 규칙 카드 — 자연어 확신도

기존 시스템은 규칙에 `[HIGH]`, `[LOW]` 같은 태그를 붙인다. 문제는 태그가 전달하지 못하는 정보가 있다는 것이다.

```
둘 다 [LOW]인 규칙:
  A: "3건 관측, 3건 일치 (100%)" → 표본 부족이지 일치율은 높음
  B: "100건 관측, 10건 일치 (10%)" → 충분한 표본에서 실제로 약한 패턴

태그는 이 차이를 구분하지 못한다.
```

규칙 카드는 자연어로 확신도를 전달한다:

```
━━━ 규칙 카드 #009 ━━━
패턴: 한진산업 대형발주(150kg+) 후 세진테크 STS316 발주
확신: 낮지만 무시 불가 — 3건뿐이나 3건 모두 일치(100%).
      표본이 적어 통계적 확신은 약하지만, 100% 일치는 우연이라 보기 어렵다.
행동: 한진산업 4월 발주량 확인 후, 150kg 이상이면 세진테크 대비.
━━━━━━━━━━━━━━━━━━
```

FUT, MetaFaith(2025) 연구에 따르면 LLM은 `[HIGH]` 같은 메타 태그보다 "거의 확실하다" 같은 hedging 표현을 더 잘 이해하고 의사결정에 반영한다.

### 차별점 4: 규칙 생애주기 — 폐기가 아닌 아카이브

대부분의 규칙 시스템은 규칙이 안 맞으면 삭제한다. 자재 발주에서는 이것이 위험하다.

```
문제: "봄에만 맞는 규칙"을 여름에 틀렸다고 삭제하면,
      다음 봄에 처음부터 다시 발견해야 한다.

해결: 4단계 건강 관리
  경고(Warning)   → 정확도 하락 감지, 확신도 낮춤
  검토(Review)    → LLM이 "왜 안 맞아졌는지" 원인 추론
  퇴역(Retire)    → 아카이브 이동 (삭제 아님)
  부활(Revive)    → "퇴역된 규칙 #4가 3회 연속 적중. 재활성화?"
```

이 설계는 Concept Drift의 4가지 유형(급변, 점진, 증분, 반복)에 맞춰 차등 대응한다. 특히 반복(Recurring) 유형 — 계절성 패턴의 일시적 소멸과 복귀 — 은 아카이브+부활 없이는 대응할 수 없다.

### 차별점 5: 검증 가능한 예측 근거

시계열 모델(LSTM 등)의 예측은 "왜 이 값인지" 설명하지 못한다.

```
LSTM: "다음 주 STS304 발주 예상: 105kg" → 왜?

이 시스템:
  "다음 주 STS304 준비 권장: 100~120kg
   근거:
   - 규칙 #001 (한진산업 월초 발주, 정확도 92%) → 100kg 기본
   - 규칙 #004 (봄 수요 30% 증가, 정확도 85%) → +20% 여유분
   - 규칙 #009 (한진 대형 후 세진 STS316, 3/3 일치) → 주시"
```

각 예측의 근거가 특정 규칙으로 추적 가능하고, 그 규칙의 정확도 이력도 확인할 수 있다. 예측이 빗나갔을 때 "어떤 규칙이 잘못되었는지" 사후 진단이 가능하다.

## 7. 정리: 왜 이 방식인가

```
기존의 선택지:

  1. 사람이 규칙 작성     → 확장 불가. 사람이 모르는 패턴은 영원히 모름.
  2. ML 모델로 예측       → 설명 불가. 왜 이 예측인지 모름. drift 시 전체 재학습.
  3. LLM에게 직접 분석    → 당연한 소리만 함. 데이터 커지면 불가.
  4. 통계 도구로 피처 추출 → 해석은 여전히 사람 몫. 혼합 패턴 미분해.

이 시스템의 선택:

  통계 도구가 구조를 잡고 (혼합 분해 포함)
  → LLM이 해석하고 ("왜?"를 추론)
  → 자연어 규칙으로 출력하고 (사람이 읽고 판단 가능)
  → 규칙이 스스로 검증받고 늙고 죽고 부활한다 (생애주기)
```

이것은 "LLM을 어떻게 잘 쓸 것인가"의 문제가 아니라, **"통계 도구와 LLM 사이의 분업을 어떻게 설계할 것인가"**의 문제다. 실험 결과가 보여주는 것은, 이 분업의 경계선 — 8-Layer 통계 스택의 설계 — 이 발견 품질을 직접 결정한다는 것이다 (v1: 6/10 → v2: 9/10).
