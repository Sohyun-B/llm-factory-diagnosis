# Step 5: 자재 발주 예측 — 살아있는 자연어 규칙 에이전트

> 작성일: 2026-03-09
> 상태: 아이디어 스케치 + 관련 연구 조사 완료

---

## 1. 한 줄 요약

> LLM이 발주 이력에서 패턴을 **자연어 규칙**으로 발견하고,
> 그 규칙이 태어나고, 검증받고, 늙고, 죽는 **생애주기**를 스스로 관리하며,
> 살아남은 규칙들로 다음 발주를 예측하는 에이전트.

---

## 2. 시나리오

나는 자재를 요청받으면 준비해서 건네주는 사람이다.

```
가지고 있는 것:
  - 자재 정보 (품목, 단가, 리드타임, 공급사)
  - 발주 이력 (누가, 언제, 뭘, 얼마나 요청했는지)
  - 배송 이력 (실제 도착일, 사고 여부, 지연 일수)

하고 싶은 것:
  "다음에 뭐가 언제 필요할지" 미리 예측해서 준비해두기
```

---

## 3. 핵심 아이디어: 자연어 규칙

예측의 근거가 ML 모델의 가중치가 아니라 **사람이 읽을 수 있는 문장**이다.

```
"한진산업은 매달 첫째 주에 STS304 100kg을 주문하는 경향이 있다."
"봄철(3~5월) STS316 가격이 평균 15% 상승한다."
"ㅁㅁ물류 운송 중 사고 발생 시 평균 5일 추가 소요된다."
"설 연휴가 끼면 한진산업 주문이 둘째 주로 밀린다."
```

이 규칙들은:
- **사용자가 직접 입력**할 수 있고 (경험 기반)
- **LLM이 데이터 분석으로 발견**할 수도 있다
- 각 규칙에 **정확도 이력**이 붙고, 시간이 지나면서 검증된다
- 잘 맞추는 규칙은 살아남고, 못 맞추는 규칙은 죽는다

---

## 4. 규칙의 생애주기

```
┌──────────────────────────────────────────────────┐
│                    탄 생                          │
│  사용자 입력 or LLM 분석으로 규칙 생성             │
│  → 자연어 문장 + 근거 + 검증 방법 정의             │
│  → 상태: 후보 (candidate)                         │
└──────────────────┬───────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────┐
│                    성 장                          │
│  예측 시점마다 맞았는지 기록                       │
│  정확도 이력 누적, 신뢰도 상승                     │
│  → 상태: 활성 (active)                            │
└──────────────────┬───────────────────────────────┘
                   │
          ┌────────┴────────┐
          ▼                 ▼
┌──────────────┐   ┌──────────────┐
│ 계속 맞음     │   │ 안 맞기 시작  │
│ → 신뢰도 상승 │   │ → 경고/검토   │
└──────────────┘   └───────┬──────┘
                           │
                  ┌────────┴────────┐
                  ▼                 ▼
          ┌──────────────┐  ┌──────────────┐
          │ 일시적 빗나감  │  │ 구조적 변화   │
          │ (계절, 이벤트) │  │ (담당자 교체) │
          │ → 유지        │  │ → 퇴역       │
          └──────────────┘  └──────┬───────┘
                                   │
                                   ▼
                          ┌──────────────┐
                          │   아카이브    │
                          │  (삭제 아님)  │
                          │  부활 가능    │
                          └──────────────┘
```

**폐기하지 않고 아카이브하는 이유**: 자재 발주는 계절성이 있다.
"봄에만 맞는 규칙"을 여름에 틀렸다고 삭제하면, 다음 봄에 다시 못 쓴다.

## 6. 규칙 발견: LLM이 복잡한 패턴을 찾게 만드는 방법

### 6-1. 문제: LLM은 당연한 소리만 한다

LLM에게 "이 데이터에서 패턴 찾아줘"라고 하면:

```
나오는 것 (쓸모없음):
  "한진산업은 자주 주문하는 고객입니다"
  "STS304가 가장 많이 팔리는 품목입니다"

원하는 것 (가치있음):
  "한진산업이 STS304를 주문한 후 평균 3일 뒤에
   대성기업이 AL6061을 주문하는 패턴이 있다
   — 같은 프로젝트 하청 관계일 가능성"
```

파인튜닝으로 해결하면 LLM 쓰는 목적이 훼손된다.
아무것도 안 하면 원론적이고 당연한 질문만 한다.

### 6-2. 해결: 큰 그림 위에서 더 깊이 파기

**"잔차를 설명해라"가 아니다.** 잔차에 집중하면 이상한 예외 케이스에 오버피팅된다.

핵심은 **큰 그림은 Phase 1에서 잡고, Phase 2에서는 그 안을 더 깊이 파는 것.**

```
비유: 아이스크림

Phase 1 (큰 그림):
  "여름에 아이스크림이 겨울보다 잘 팔린다"
  → 이건 당연하고, 이것만으로는 쓸모없음

Phase 2 (깊이 파기):
  "여름 중에서도 유독 평일 오후 4시에 잘 팔린다. 왜?"
  → 중고등학생 수업 종료 시간이기 때문
  → 이게 insight
```

Phase 2는 Phase 1에 "대비되는 예외"를 찾는 게 아니라,
**Phase 1의 큰 패턴 안에서 더 구체적인 "왜"를 파고드는 것.**

```
Phase 1 — 큰 그림 잡기 (통계 도구)
  ├ 고객별 발주 주기
  ├ 품목별 계절성
  ├ 요일/월별 분포
  └ 결과: "이미 알고 있는 큰 패턴 목록"

Phase 2 — 큰 그림 안에서 깊이 파기 (도구 + LLM)
  큰 패턴을 전제로 깔고, 그 안의 구조를 탐색:

  "한진산업이 매달 주문하는 건 알겠다.
   그런데 왜 어떤 달은 첫째 주이고 어떤 달은 둘째 주인가?"
  → 공휴일 영향? 월초 영업일 수? 전월 발주량과의 관계?

  "STS304가 인기 품목인 건 알겠다.
   그런데 같이 주문되는 품목 조합이 있는가?"
  → 품목 간 동반 발주 패턴 (장바구니 분석)

  "봄에 수요가 오르는 건 알겠다.
   그런데 어떤 고객이 먼저 움직이고 어떤 고객이 따라오는가?"
  → 선행 고객 발견 → 선행 지표로 활용 가능
```

### 6-3. 5단계 질문 깊이

| Level | 누가 | 하는 것 | 예시 |
|-------|------|---------|------|
| **0** | 도구 | 기초 통계 | "한진산업 평균 발주 주기: 28일" |
| **1** | 도구 → LLM 정리 | 큰 그림 패턴 | "한진산업은 매달 STS304 100kg 주문" |
| **2** | 도구 + LLM | 큰 그림 안의 "왜" | "그 중 첫째 주 발주가 80%인 이유: 한진산업 월초 구매 승인 프로세스 때문 추정" |
| **3** | 도구 + LLM | 교차 패턴 | "한진산업 발주 후 3~5일 내 대성기업 발주 확률 78% → 원청/하청 관계 가능성" |
| **4** | LLM | 맥락 추론 | "최근 한진산업 발주량이 늘고 STS316 문의도 시작 → 새 프로젝트, 향후 수요 증가 예상" |

**Level 0~1은 도구가, Level 2~4는 LLM이 담당.**
핵심은 "예외를 설명해라"가 아니라 **"이 패턴의 원인이 뭘까"**, **"이 패턴 안에 더 세부 구조가 있을까"**를 시키는 것.

### 6-4. LLM에게 주는 분석 프롬프트 구조

```
[큰 그림 — 이미 알려진 패턴]
다음은 통계 분석으로 확인된 기본 패턴입니다:
- 한진산업: STS304, 월 1회, 평균 100kg, 주기 28±3일
- 봄철(3~5월): 전체 발주량 연평균 대비 130%
- 대성기업: 격주 발주 경향 (검증 중)

[큰 그림 안에서 도구가 발견한 세부 단서]
- 한진산업 발주 요일 분포: 월 68%, 화 20%, 기타 12%
- 한진산업 발주 후 3~5일 내 대성기업 발주 동반 빈도: 78% (우연 기대값 30%)
- 봄철 수요 증가가 가장 먼저 나타나는 고객: 한진산업 (2월 말부터)
- STS304와 AL6061이 같은 주에 발주되는 빈도: 65%

[질문]
위 단서들을 보고:
1. 왜 이런 패턴이 존재하는지 추론해라 (원인 가설)
2. 가설이 맞다면 앞으로 예측에 어떻게 쓸 수 있는지 규칙으로 만들어라
3. 무리하게 규칙을 만들지 마라 — 설명할 수 없으면 "추가 관찰 필요"로 표시
```

### 6-5. 도메인별 질문 템플릿 (파인튜닝 대신)

파인튜닝 없이, **"어디를 볼지" 방향만 잡아주는 분석 프레임워크**:

```
교차 패턴:
  "고객 A의 발주 후 N일 이내에 고객 B가 발주하는 경우가
   우연보다 유의미하게 많은 조합이 있는가?"
  → 도구: 모든 고객 쌍의 시간차 상관관계 계산
  → LLM: 유의미한 결과에 대해 "왜?" 추론

수량 이상:
  "평소 발주량 대비 ±50% 이상인 건들의 공통점은?"
  → 도구: 이상 발주 필터링 + 시점/고객/품목 교차 분석
  → LLM: 공통 맥락 해석

선행 지표:
  "발주 전 N일 이내에 가격 문의, 샘플 요청, 견적 요청이
   있었던 경우의 비율은?"
  → 도구: 이벤트 선후 관계 통계
  → LLM: 선행 지표로 쓸 수 있는지 판단

계절 전환:
  "작년 같은 시기 대비 올해 발주 패턴이 달라진 고객은?"
  → 도구: YoY 비교 통계
  → LLM: "왜 달라졌는가" 가설 생성
```

이 템플릿 자체는 사람이 한 번만 만들면 된다.
파인튜닝이 아니라 **분석 방향을 구조화**하는 것.

### 6-6. 정리

```
LLM에게 "패턴 찾아줘"         → ❌ 당연한 소리만 함
LLM을 파인튜닝                → ❌ LLM 쓰는 목적 훼손
잔차(예외)만 보여주기          → ❌ 이상한 것에 오버피팅

정답:
  1. 통계 도구가 큰 그림(기본 패턴)을 먼저 잡음
  2. 큰 그림 안에서 세부 단서를 도구가 추출
  3. LLM은 그 단서의 "왜"를 추론하고 insight를 규칙화
  4. 도메인별 질문 템플릿으로 "어디를 볼지" 방향만 잡아줌
  → 파인튜닝 없이도 복잡한 규칙 발견 가능
```

## 7. 규칙 건강 관리: Concept Drift 감지

"3연속 미스면 폐기"는 너무 단순하다.
규칙이 안 맞아지는 이유가 4가지로 다르기 때문:

| 유형 | 예시 | 감지 방법 | 대응 |
|------|------|-----------|------|
| **급변** (Sudden) | 담당자 교체로 패턴 완전 변경 | 최근 N회 연속 미스 | 퇴역 + 원인 조사 |
| **점진** (Gradual) | 발주 주기가 서서히 길어짐 | 이동평균 정확도 하락 추세 | 규칙 수정 ("2주 → 3주") |
| **증분** (Incremental) | 봄 시즌이 점점 빨라짐 | 적중하나 시점이 밀림 | 시간 파라미터 조정 |
| **반복** (Recurring) | 계절성 패턴이 잠시 사라졌다 복귀 | 시즌 태그 + 아카이브 보존 | 폐기하지 않고 대기 |

### 4단계 건강 관리

```
Level 1 — 경고 (Warning):
  ADWIN 기준 최근 정확도가 유의미하게 하락
  → 예측에 반영하되 확신도를 낮춤
  → 규칙 카드에 "⚠️ 추세: 하락 중" 표시

Level 2 — 검토 (Review):
  정확도 50% 미만 + 검증 5회 이상
  → LLM이 "왜 안 맞아졌는지" 원인 추론
  → 수정 가능하면 규칙 업데이트, 아니면 퇴역 권고

Level 3 — 퇴역 (Retire):
  정확도 30% 미만 or 검토 후 원인 불명
  → 아카이브 이동 (삭제 아님)

Level R — 부활 (Revive):
  아카이브 규칙이 최근 다시 적중
  → "폐기된 규칙 #4가 3회 연속 적중. 재활성화할까요?"
```

## 8. 규칙 카드: LLM에게 신뢰도를 전달하는 방법

### 문제

규칙마다 정확도가 다른데, LLM에게 이걸 어떻게 전달해야 차등 대우하는가?

### `[HIGH]` 태그보다 나은 방법: 자연어 확신도

2025년 연구(FUT, MetaFaith)가 보여준 것:
- LLM은 `[HIGH]` 같은 메타 태그보다 "거의 확실하다" 같은 hedging 표현을 더 잘 이해
- 심리언어학 기반 9단계 확률-언어 매핑이 존재

```
확률 구간    →  자연어 표현
[0.0, 0.2]  →  "거의 없다"
[0.2, 0.4]  →  "가능성이 낮다"
[0.4, 0.6]  →  "불확실하다"
[0.6, 0.8]  →  "가능성이 높은 편이다"
[0.8, 1.0]  →  "거의 확실하다"
```

### 규칙 카드 포맷

자연어 확신도 + 근거 수치 + 최근 추세 + 행동 지침을 한 덩어리로:

```
━━━ 규칙 카드 #001 ━━━
패턴: 한진산업은 매달 첫째 주에 STS304 100kg을 주문한다.
발견: 2026-01-15 (LLM 분석)
검증: 12회 중 11회 적중 (92%)
최근: 1월 ✓, 2월 ✓, 3월 ✓ (3연속 적중)
추세: 안정적
확신: 거의 확실하다. 이 패턴에 기반한 준비를 권장한다.
예외: 설 연휴가 끼면 둘째 주로 밀림 (규칙 #007 참조)
━━━━━━━━━━━━━━━━━━

━━━ 규칙 카드 #009 ━━━
패턴: 대성기업은 격주로 AL6061을 발주한다.
발견: 2026-02-10 (LLM 분석)
검증: 5회 중 3회 적중 (60%)
최근: 2월3주 ✗, 2월5주 ✓, 3월1주 ✗ (불안정)
추세: 하락 중 ⚠️
확신: 가능성은 있으나 불확실하다. 참고만 하고 단독 근거로 쓰지 말 것.
━━━━━━━━━━━━━━━━━━
```

**왜 태그보다 이게 나은가:**
1. "거의 확실하다"는 문맥에 녹아들어 LLM이 무시하기 어려움
2. "12회 중 11회"라는 구체적 수치가 판단 근거로 작동
3. "단독 근거로 쓰지 말 것"이라는 행동 지침이 자연스럽게 포함
4. "추세: 하락 중"으로 LLM이 스스로 경계 가능

## 9. 에이전트 구조

```
사용자: "다음 주에 준비해야 할 자재가 뭐가 있을까?"

Orchestrator
  │
  ├─→ [도구] 이력 조회
  │     과거 같은 시기 발주 이력 조회
  │     → 테이블 반환
  │
  ├─→ [RAG] 규칙 검색
  │     "다음 주"에 적용 가능한 활성 규칙 검색
  │     → 규칙 카드 3~5개 반환
  │
  ├─→ [도구] 외부 맥락
  │     공휴일, 시즌, 특수 이벤트 확인
  │     → "3월 둘째 주, 특이 이벤트 없음"
  │
  └─→ [LLM] 종합 판단
        이력 + 규칙 카드 + 맥락을 보고:

        "준비 권장:
         1. STS304 100kg — 규칙 #001 기반, 거의 확실 (92%)
         2. AL6061 50kg — 이력 기반 추정, 가능성 있음

         확인 필요:
         3. 대성기업에 다음 주 발주 계획 문의 권장
            (규칙 #009의 정확도가 하락 중이라 확신 부족)"
```

## 10. 기존 연구와의 비교

### 있는 것 (기존 연구에서 확인)

| 개념 | 출처 | Step 5와의 관계 |
|------|------|----------------|
| 가설 생성→검증→폐기 사이클 | HypoGeniC (Peirce 프레임워크) | 동일한 구조 |
| 가설에 베이지안 확률 부여 | HypoAgents (2025) | 정확도 관리에 참조 가능 |
| RAG 응답 신뢰도 점수 | Cleanlab TLM | 응답 품질 관리에 참조 가능 |
| 자연어 불확실성 표현 | FUT, MetaFaith (2025) | 규칙 카드 확신도 표현에 직접 적용 |
| 비즈니스 규칙 버전/감사 관리 | BRMS (Drools 등) | 규칙 거버넌스 구조에 참조 |
| Concept Drift 감지 | DDM, ADWIN 등 | 규칙 건강 관리에 직접 적용 |

### 없는 것 (Step 5의 독자적 기여 가능성)

| 개념 | 설명 |
|------|------|
| **규칙별 검증 방법 자동 정의** | LLM이 규칙 생성 시 "어떻게 확인할지"도 함께 정의 |
| **규칙별 정확도 이력 추적** | 시간에 따른 개별 규칙의 성적표 |
| **자연어 확신도 차등 전달** | 규칙 카드로 LLM에 신뢰도를 자연어로 전달 |
| **규칙 퇴역/부활 자동화** | Concept Drift 감지 + 계절성 고려 아카이브 |
| **사용자 규칙 + LLM 규칙 통합** | 경험적 지식과 데이터 기반 발견의 공존 |
| **예측 근거의 완전한 추적** | "이 예측은 규칙 #001, #007 때문" — 사후 검증 가능 |

### LLM 귀납 추론의 한계 (주의)

LLM은 "강한 귀납 추론자는 아니다" (ACL 2025):
- 잘함: 자연어 맥락의 경향성 포착, 상식 기반 추론, 관찰 요약
- 못함: 노이즈 데이터에서 정확한 수치 규칙 추출

→ **통계 도구가 패턴을 찾고, LLM이 자연어로 해석하는 분업이 필수**

### 산업 BRMS와의 차이

```
BRMS(Drools 등)가 하는 것:
  ✅ 규칙 버전 관리, 감사 추적, 런타임 추가/삭제, 충돌 감지

BRMS가 못 하는 것 (= Step 5가 자동화하는 것):
  ❌ 규칙이 아직 맞는지 자동 검증
  ❌ 새 규칙을 데이터에서 자동 발견
  ❌ 규칙의 정확도 추적
  → 현재는 전부 사람이 주기적으로 리뷰
```
