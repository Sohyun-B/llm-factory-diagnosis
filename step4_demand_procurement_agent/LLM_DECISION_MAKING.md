# LLM의 의사판단: 비즈니스 로직을 어떻게 주고, 얼마나 복잡한 판단이 가능한가

> 작성일: 2026-03-09
> 전제: LLM은 나쁜 계산기지만 괜찮은 의사결정자. 계산은 도구가 하면 됨.
> 질문: **의사판단 모델을 LLM에게 어떻게 주는가? 얼마나 복잡한 판단이 가능한가?**

---

## 1. 비즈니스 로직을 LLM에게 주는 4가지 방식

### 방식 1: System Prompt — 규칙을 텍스트로 직접 써준다

```
시스템 프롬프트:
"너는 재고 관리 담당자다.
 - 안전재고 = 평균 주간 수요 × 리드타임(주) × 1.5
 - 재주문점(ROP) = 안전재고 + (평균 주간 수요 × 리드타임)
 - 현재 재고가 ROP 아래면 발주를 권장해라
 - 발주량 = EOQ 공식으로 계산 도구를 호출해라
 - 공급사가 2개 이상이면 리드타임이 짧은 쪽을 우선 선택해라"
```

**장점**: 간단, 빠르게 변경 가능
**한계**: 규칙이 많아지면 프롬프트가 길어져서 LLM이 일부를 무시하기 시작
**실제 사례**: BPMN 모델링 시스템 — system prompt + one-shot 예시로 비즈니스 프로세스 규칙 전달

### 방식 2: Knowledge Graph / RAG — 규칙을 검색 가능한 구조로 저장

```
비즈니스 규칙 DB (벡터 DB 또는 Knowledge Graph):
  ┌─────────────────────────────────────────────┐
  │ 규칙: "계절 품목은 시즌 3개월 전에 발주"        │
  │ 규칙: "냉동식품은 최소 발주량 500kg"            │
  │ 규칙: "A등급 공급사 납기 준수율 > 95%"          │
  │ 규칙: "원자재 가격이 기준가 대비 120% 초과시     │
  │        대체 공급사 탐색"                        │
  └─────────────────────────────────────────────┘

LLM이 상황에 따라 관련 규칙을 검색(retrieve)해서 판단에 활용
```

**장점**: 규칙이 수백 개여도 관리 가능, 상황별로 필요한 것만 가져옴
**한계**: 검색 품질에 의존 — 관련 규칙을 못 찾으면 모르는 채로 판단
**실제 사례**:
- 제약 (Pharmacovigilance) — 약물 상호작용 KG + SNOMED-CT 의학 용어 체계
- 섬유 디자인 — DIN 표준, EU 규제, 소재 특성 DB를 에이전트가 검색

### 방식 3: Tool Definition — 판단 로직을 도구 설명에 녹인다

```python
tools = [
    {
        "name": "calculate_reorder_point",
        "description": "재주문점 계산. 현재 재고가 이 값 아래면 발주 필요.",
        "parameters": {
            "avg_weekly_demand": "최근 12주 평균 주간 수요",
            "lead_time_weeks": "공급사 평균 리드타임(주)",
            "service_level": "목표 서비스 수준 (기본 0.95)"
        }
    },
    {
        "name": "evaluate_supplier",
        "description": "공급사 평가. 납기준수율, 품질불량률, 가격경쟁력을 종합 점수로 반환.",
        "parameters": {
            "supplier_id": "공급사 코드",
            "period": "평가 기간 (기본: 최근 6개월)"
        }
    }
]

# LLM은 도구의 "description"을 읽고
# 언제 어떤 도구를 호출할지 스스로 결정
```

**장점**: 비즈니스 로직이 도구 안에 캡슐화됨 — LLM은 "언제 호출할지"만 판단
**한계**: 도구 설명이 애매하면 LLM이 잘못된 상황에서 호출
**실제 사례**: JD.com — 기존 검증된 예측 함수/분석 함수를 function calling으로 호출

### 방식 4: Routine / Workflow — 의사결정 절차 자체를 구조화

```
Routine: 월간 원자재 발주 프로세스
├── Step 1: 수요예측 도구 호출 → 다음 달 예상 수요 산출
├── Step 2: 현재 재고 조회 → 부족분 계산
├── Step 3: 부족분 > 0이면
│   ├── Step 3-1: 공급사별 견적 조회
│   ├── Step 3-2: 리드타임 + 가격 + 품질 종합 평가
│   └── Step 3-3: 최적 공급사 + 발주량 제안
├── Step 4: 부족분 ≤ 0이면
│   └── "발주 불필요. 현재 재고로 N주 커버 가능" 보고
└── Step 5: 판단 근거를 자연어로 설명
```

**장점**: LLM이 임의로 건너뛰지 못하도록 절차를 강제
**한계**: 예상 못한 상황(긴급 발주, 공급 중단)에 대한 유연성 부족
**실제 사례**: "Routine" 프레임워크 (arXiv 2507.14447) — 전체 워크플로우를 명시적으로 인코딩

## 2. 4가지 방식의 비교

| 방식 | 복잡한 규칙 처리 | 유연성 | 안정성 | 적합한 상황 |
|------|----------------|--------|--------|------------|
| System Prompt | 규칙 ~10개까지 | 높음 | 낮음 (무시 가능) | 프로토타입, 단순 판단 |
| KG/RAG | 수백 개 가능 | 중간 | 중간 (검색 품질 의존) | 규칙이 많고 상황별로 다른 경우 |
| Tool Definition | 도구 수에 비례 | 높음 | 높음 (로직이 코드 안) | 계산/조회가 핵심인 경우 |
| Routine/Workflow | 절차 복잡도에 비례 | 낮음 | 높음 (절차 강제) | 정해진 프로세스를 따라야 하는 경우 |

**실무 최적 조합**: Routine(절차 골격) + Tool(계산 위임) + RAG(상황별 규칙 검색)

## 3. LLM 의사판단의 복잡도 수준

### Level 1: 단순 분류 — "이건 A인가 B인가"

```
입력: "현재 재고 500개, ROP 600개"
판단: "재고가 ROP 아래이므로 발주 필요"
```
→ if문 하나로 충분. LLM을 쓸 이유가 거의 없음.

### Level 2: 다변수 판단 — "여러 요소를 종합해서 결정"

```
입력:
  - 현재 재고: 500개
  - 다음 달 예상 수요: 800개 (시계열 모델 결과)
  - 공급사 A: 가격 저렴, 리드타임 4주, 최근 납기 지연 3건
  - 공급사 B: 가격 10% 비쌈, 리드타임 2주, 납기 지연 0건
  - 시즌 요인: 다음 달 성수기

판단: "성수기라 수요 변동이 클 수 있으므로 리드타임이 짧은 B 공급사에
      우선 발주하되, 수량의 30%는 가격이 저렴한 A에서 조달 권장"
```
→ **이게 LLM이 잘하는 영역**. 정형화하기 어려운 다변수 트레이드오프를 자연어로 추론.
→ 기존 방법: 가중 점수 모델(weighted scoring) — 가중치를 사람이 정해야 함
→ LLM: 상황 맥락을 읽고 유연하게 가중치를 조절

### Level 3: 다단계 추론 — "이전 판단이 다음 판단에 영향"

```
Step 1: 수요예측 → "다음 달 수요 800개, 신뢰구간 [650, 950]"
Step 2: 신뢰구간이 넓다 → "불확실성 높음" → 안전재고 상향 결정
Step 3: 안전재고 상향 → 발주량 증가 → 예산 초과 가능성
Step 4: 예산 제약 확인 → "예산 내에서 가능한 최대 물량은 700개"
Step 5: 700개로 부족할 경우 대응 방안 → "스팟 구매 옵션 확보"
```
→ 각 단계의 결과가 다음 단계의 입력이 됨
→ **Planning → Execution → Observation → Replanning 루프** (JD.com 방식)
→ 이때 핵심은 LLM이 중간 결과를 보고 **계획을 수정**할 수 있다는 것

### Level 4: 불확실성 하에서의 판단 — "모르는 걸 알고 행동"

```
상황: 공급사 C가 갑자기 납기를 "미정"으로 변경
LLM이 해야 할 것:
  1. "이 정보가 신뢰할 만한가?" → 과거 패턴 확인 (RAG)
  2. "대체 공급사가 있는가?" → 공급사 DB 검색 (Tool)
  3. "긴급도는?" → 현재 재고 대비 소진 예상일 계산 (Tool)
  4. 확신이 없으면 → "담당자에게 확인 필요" 보고 (Human-in-the-loop)
```
→ **LLM이 "모르겠다"고 말할 수 있어야 함** — 이것이 가장 어려움
→ 불확실성 정량화(Uncertainty Quantification)와 결합해야 신뢰도 확보

## 4. 핵심 인사이트: LLM 의사결정이 진짜 잘 되려면

### 원칙 1: LLM에게 "왜"를 주고, "어떻게"는 도구에 맡겨라

```
나쁜 설계:
  프롬프트: "EOQ = sqrt(2DS/H)로 계산해서 발주량 알려줘"
  → LLM이 수학을 틀릴 수 있음

좋은 설계:
  프롬프트: "발주가 필요한지 판단해. 필요하면 calculate_eoq 도구를 호출해."
  도구: calculate_eoq(demand, setup_cost, holding_cost) → 정확한 계산 결과 반환
  → LLM은 "발주가 필요한가?"만 판단, 숫자는 도구가 계산
```

### 원칙 2: 판단 근거를 반드시 출력하게 하라

```
나쁜 출력:
  "공급사 B에 700개 발주하세요."

좋은 출력:
  "공급사 B에 700개 발주를 권장합니다.
   근거:
   1. 다음 달 예상 수요 800개 중 현재 재고 150개 차감 → 부족분 650개
   2. 안전재고 50개 추가 → 총 필요 700개
   3. 공급사 B 선택 이유: 성수기 리드타임 리스크 고려, B의 2주 리드타임이
      A의 4주보다 안전 (과거 성수기 A사 납기 지연율 15%)
   4. 비용 차이: A 대비 10% 높지만 결품 시 기회비용 고려하면 합리적"
```
→ 사람이 검토하고 승인할 수 있게 됨
→ 틀렸을 때 "어디서 틀렸는지" 추적 가능

### 원칙 3: 모르는 건 모른다고 하게 설계하라

```
시스템 프롬프트에 포함:
"다음 상황에서는 판단을 보류하고 담당자 확인을 요청하라:
 - 예측 신뢰구간이 평균 대비 ±40% 이상일 때
 - 공급사 정보가 3개월 이상 업데이트되지 않았을 때
 - 과거에 없던 새로운 품목일 때"
```
→ **과신 방지**가 LLM 의사결정의 가장 중요한 설계 요소

### 원칙 4: System 1 / System 2를 구분하라

```
System 1 (빠른 판단 — 규칙 기반):
  "재고 < ROP → 자동 발주 트리거"
  → LLM이 개입할 필요 없음. 규칙 엔진이 처리.

System 2 (느린 판단 — LLM 추론):
  "이번 달 수요가 비정상적으로 높은데, 일시적인가 추세인가?"
  "공급사 A가 최근 품질 이슈가 있었는데, 비중을 줄여야 하나?"
  → 맥락을 읽고 종합적으로 추론해야 하는 상황
  → 여기에 LLM을 쓴다
```
→ **모든 의사결정에 LLM을 쓰지 않는다**
→ 정형화 가능한 판단은 규칙 엔진, 비정형 판단만 LLM

## 5. 실제 산업에서 동작하는 패턴 (논문 기반)

### 패턴 A: 제조 현장 — ML 예측 + LLM 해석 (Trustworthy Process Management)

```
1. MES(제조실행시스템) 이벤트 로그 → ML 모델이 작업 소요시간 예측
2. 불확실성 정량화: Monte Carlo Dropout으로 신뢰구간 산출
   → "절삭 공정: 8.2시간 ± 1.1시간"
3. XAI(설명가능 AI): Shapley value로 어떤 변수가 예측에 영향 줬는지
4. LLM이 이 모든 것을 종합해서 자연어로 설명
   → "절삭 공정이 평소보다 길어질 수 있습니다.
      주요 원인: 소재 경도가 높은 배치(Shapley 기여도 0.35).
      권장: 공구 교체 주기를 앞당기세요."
```

**LLM의 역할**: 숫자 결과를 "왜 그런지" 해석하고 "뭘 해야 하는지" 제안
**핵심**: 예측/계산은 전부 ML+통계가 하고, LLM은 해석+권장만

### 패턴 B: 제약 안전성 — KG 기반 의사결정 (Pharmacovigilance)

```
1. 부작용 보고서 입력 (비정형 텍스트)
2. LLM이 보고서에서 약물명, 증상, 인과관계 추출
3. Knowledge Graph 조회: 해당 약물의 알려진 부작용 목록
4. LLM 판단:
   - 이 부작용이 이미 알려진 것인가? → KG에서 확인
   - 새로운 부작용 신호인가? → 유사 사례 검색(RAG)
   - 심각도 분류 → 규제 보고 필요 여부 결정
5. 최종 출력: 구조화된 보고서 + 판단 근거
```

**LLM의 역할**: 비정형 텍스트 이해 + KG와 교차 검증 + 분류 판단
**핵심**: KG가 "사실의 근거"를 제공하여 LLM의 환각을 억제

### 패턴 C: TimeSeriesScientist — 멀티모달 판단

```
1. 시계열 데이터 → 통계 요약 + 시각화(플롯) 생성 (코드가 수행)
2. LLM이 플롯 이미지를 직접 보고:
   - "뚜렷한 주간 계절성 있음" (시각적 패턴 인식)
   - "2024년 Q3에 레벨 시프트 발생" (변화점 감지)
3. 이 시각적 판단을 기반으로:
   - 적합한 모델 후보 선택 (21개 모델 라이브러리에서)
   - 전처리 전략 결정 (IQR vs Z-score 이상치 처리)
4. 모델 학습 + 검증은 코드가 수행
5. LLM이 검증 결과 보고 앙상블 전략 결정
```

**LLM의 역할**: 시각적 패턴 인식(멀티모달) + 전략 선택
**핵심**: 사람 데이터 사이언티스트가 "그래프 보고 감 잡는" 행위를 LLM이 대행

## 6. 정리: 의사판단의 질을 높이는 설계 체크리스트

| # | 설계 원칙 | 구체적 행동 |
|---|-----------|------------|
| 1 | 계산과 판단을 분리 | 숫자 계산 = 도구, 맥락 해석 = LLM |
| 2 | 비즈니스 로직 구조화 | 규칙 10개 이하 = 프롬프트, 그 이상 = RAG/KG |
| 3 | 절차 강제 | 정해진 프로세스 = Routine/Workflow로 구조화 |
| 4 | 판단 근거 출력 필수 | 모든 추천에 "왜" 포함. 사후 검증 가능 |
| 5 | 과신 방지 | 불확실할 때 보류하는 규칙 명시 |
| 6 | System 1/2 구분 | 정형 판단 = 규칙 엔진, 비정형 판단 = LLM |
| 7 | 도구 설명 정밀 작성 | 도구의 description이 곧 LLM의 판단 기준 |
| 8 | 멀티모달 활용 | 시각화/차트를 LLM에 보여주면 패턴 인식력 향상 |
