# Step 4: 수요예측 & 원자재 조달 — LLM은 실제로 "어떻게" 쓰이는가

> 작성일: 2026-03-09
> 초점: 성능 수치가 아닌 **메커니즘** — LLM이 구체적으로 무엇을 하는지

---

## 1. 핵심 질문

> "LLM 쓰니까 좋아졌다"는 안 믿는다.
> **LLM이 수요예측/조달에서 구체적으로 어떤 방식으로 쓰이는가?**
> 기존에 없던 혁신적인 사용 방식이 있는가?

---

## 2. 논문별 실제 메커니즘 분석

### Paper 1: JD.com — Supply Chain Planning (arXiv 2509.03811)

**LLM이 실제로 하는 것: 코드 생성 + 도구 호출 오케스트레이터**

LLM은 수요를 직접 예측하지 않는다. 대신:

```
사용자 질문 (자연어)
  → LLM이 Text-to-SQL로 DB에서 데이터 추출
  → LLM이 pandas 코드를 한 줄씩 생성 (Filter → Transform → Groupby → Sort)
  → 기존 예측 함수/분석 함수를 function calling으로 호출
  → 결과를 요약된 텍스트로 받아서 해석
  → 다음 단계를 재계획 (closed-loop)
```

**핵심 설계 원칙: LLM은 수학을 하지 않는다**
- 숫자 계산: pandas 코드가 실행
- 예측 모델: 기존 검증된 함수를 호출 (LLM이 학습하거나 추론하지 않음)
- LLM이 받는 입력: raw 숫자가 아닌 **집계된 요약** ("매출 추이: 안정적 성장, 휴일 피크")
- LLM이 내리는 결정: "프로모션 예산 150% 증가" 같은 **정성적 의사결정**

**혁신적인 점:**
- 코드를 한 번에 다 생성하지 않고 **atomic operation 단위로 한 줄씩** 생성
- 각 단계 결과를 보고 다음 단계를 **재계획** (Planning → Execution → Observation → Replanning)
- 기존 MIP(혼합정수계획) 솔버처럼 한 번에 최적화하는 게 아니라 **반복적으로 분해하고 재조립**

**솔직한 평가:**
LLM은 여기서 "자연어 인터페이스 + 코드 생성기 + 태스크 분해기" 역할이다.
수요예측 자체를 LLM이 하는 게 아니라, 기존 모델/함수를 조합하는 오케스트레이터.

### Paper 2: Agentic LLMs — Multi-Agent Consensus (arXiv 2411.10184, IJPR 2025)

**LLM이 실제로 하는 것: 공급망 기업 역할극 + 협상**

각 LLM 에이전트가 공급망의 한 기업(소매상/도매상/제조사)을 맡아서
**자연어로 재고 수준, 납기를 "협상"**한다.

```
Agent A (소매상): "다음 주 수요가 200개 예상됩니다. 150개 발주합니다."
Agent B (도매상): "재고가 100개뿐입니다. 120개까지 가능합니다."
Agent A: "그러면 나머지 30개는 언제 받을 수 있나요?"
Agent B: "3일 후 입고 예정입니다. 우선 120개 출하하겠습니다."
→ 합의 도달
```

**핵심: 각 에이전트가 "이기적 목표 + 전체 최적"을 동시에 추구**
- 기존 방법: 중앙 집중식 최적화 (모든 정보를 한 곳에 모아서 MIP 풀기)
- LLM 방법: 분산된 에이전트가 **제한된 정보로 대화하며** 합의
- 이건 실제 공급망과 비슷함 — 기업들은 서로의 재고를 모르고, 협상으로 조율

**혁신적인 점:**
- 기존 멀티 에이전트 시뮬레이션(ABS)은 규칙 기반 — "재고 < 안전재고면 발주"
- LLM 에이전트는 **규칙 없이 상황을 읽고 유연하게 대응**
- 채찍효과(bullwhip) 완화: 적절한 합의 프레임워크를 주면 규칙 기반보다 나음

**솔직한 평가:**
진짜 흥미로운 접근이지만, 결국 LLM에 역할극을 시킨 것.
"LLM이 협상을 잘할까?"라는 질문에 대한 실험이지, 실무 배포는 아직 먼 이야기.

### Paper 3: DCATS — Data-Centric Agent for Time Series (arXiv 2508.04231)

**LLM이 실제로 하는 것: "어떤 보조 데이터를 학습에 포함할지" 선택**

LLM이 예측 모델을 만드는 게 아니다. 이미 있는 예측 모델에 **어떤 데이터를 먹일지** 결정한다.

```
교통량 예측 시나리오:
- 목표: 센서 A의 내일 교통량 예측
- 선택지: 주변 센서 B, C, D, E... 중 어떤 것을 같이 학습에 포함?
  → LLM이 메타데이터(도로 거리, 시간대 유사성, 지리적 근접성) 보고 선택
  → 5개 조합 제안 → 각각 학습 → 결과 보고 → 다시 제안 (반복)
```

**실제 데이터 클리닝은 LLM이 안 함:**
- 이상치 제거: discord-based anomaly detection (자동, LLM 무관)
- LLM은 "이웃 센서 선택"이라는 **피처 엔지니어링 의사결정**만 담당

**솔직한 평가:**
제목은 "data-centric"이지만 실제로 LLM이 하는 건 **neighbor selection** 한 가지.
기존 수동 피처 엔지니어링을 LLM 프롬프트로 대체한 것.
수요예측에 적용하면: "어떤 품목의 판매 데이터를 같이 참고할지" LLM이 고르는 정도.

### Paper 4: TimeSeriesScientist (arXiv 2510.01538)

**LLM이 실제로 하는 것: AutoML 파이프라인의 의사결정자**

4개 에이전트가 순차적으로 작동:

```
Curator Agent:
  - 통계 요약(평균, 표준편차, 추세) 계산은 코드가 함
  - LLM은 시각화(플롯 이미지)를 보고 "계절성 있음", "이상치 있음" 판단
  - 전처리 방법 선택 (IQR vs Z-score, 보간법 선택)

Planner Agent:
  - 21개 모델 라이브러리(통계/ML/딥러닝)에서 데이터 특성에 맞는 모델 선택
  - LLM이 플롯의 시각적 패턴을 읽고 → 모델 후보군 매핑
  - 하이퍼파라미터 탐색은 코드가 실행

Forecaster Agent:
  - 앙상블 전략 선택 (single best / 가중평균 / robust aggregation)
  - LLM이 검증 결과 보고 전략 결정

Reporter Agent:
  - 전체 과정을 자연어로 설명 (왜 이 모델을 골랐는지 등)
```

**혁신적인 점:**
- LLM이 **시각화 이미지**를 multimodal로 읽어서 시계열 패턴 판단
- 사람 데이터 사이언티스트의 워크플로우를 모방: 진단 → 계획 → 실행 → 보고

**솔직한 평가:**
가장 체계적인 접근. 하지만 LLM은 여전히 "의사결정자"일 뿐, 계산은 외부 도구가 함.
기존 AutoML(Auto-sklearn, FLAML 등)과의 차이는 **자연어 설명 가능성**뿐일 수 있음.

### Paper 5: AIM-Bench — LLM의 재고 의사결정 편향 (arXiv 2508.11416)

**LLM이 실제로 하는 것: Newsvendor / Beer Game에서 매 기간 발주량 결정**

```
매 기간 LLM이 받는 정보:
  - 현재 재고 수준 (I_t)
  - 과거 수요 실현값 (D_t-1, D_t-2, ...)
  - 이동 중인 주문 상태
  - 리드타임

LLM의 출력: "이번 기간에 몇 개 발주할 것인가" (자연어 → 숫자)
```

**발견된 문제 (이게 이 논문의 진짜 기여):**

| 편향 | 내용 | 원인 |
|------|------|------|
| **평균 앵커링** | LLM이 최적 발주량이 아닌 평균 수요에 가까운 값을 낸다 | 학습 데이터의 통계적 패턴 매칭 |
| **채찍효과** | 상류 에이전트일수록 주문 변동성 증폭 | 수요를 과대 추정하는 경향 |
| **프레이밍 효과** | "이익 최대화"와 "손실 최소화" 프레이밍에 따라 결과가 달라짐 | 프롬프트 워딩에 민감 |

**완화 방법:**
- 인지적 성찰: "System 2 사고를 모방하라"는 프롬프트 추가 → 앵커링 감소
- 정보 공유: 상하류 파트너의 재고/주문 정보를 LLM에 제공 → 채찍효과 감소

**솔직한 평가:**
이 논문이 가장 정직하다. "LLM에게 재고 결정 시키면 이런 편향이 있다"는 것을 보여줌.
실무적으로 가장 중요한 시사점: **LLM에 숫자 의사결정을 맡기면 편향이 생긴다.**

### Paper 6: SIR — Socratic Iterative Reasoning (ICIS 2025)

**LLM이 실제로 하는 것: Beer Game에서 "자문자답" 반복으로 발주 결정 개선**

```
기존 CoT (Chain-of-Thought):
  "수요가 증가 추세이므로... 따라서 150개 발주" (한 번에 추론)

SIR (Socratic Iterative Reasoning):
  Round 1: "현재 재고는? 200개. 지난 4주 수요 평균은? 180개."
  Round 2: "리드타임 고려하면? 4주 후 입고. 안전재고는?"
  Round 3: "상류 파트너 상황은? 이전 주문이 아직 이동 중."
  Round 4: "최종 결정: 160개 발주"
  → 반복적 자기 질문으로 더 신중한 결정
```

**vs 다른 방법:**
- RAG: 과거 유사 사례를 검색해서 참고 → 지식 통합이 어려움
- CoT: 선형적 추론 → 복잡한 다단계 공급망에서 한계
- SIR: **구조화된 대화로 추론 경로를 강제** → 특히 상류(도매/제조)에서 효과

**솔직한 평가:**
결국 프롬프트 엔지니어링의 변형이다. 하지만 "어떤 구조의 프롬프트가
공급망 의사결정에 효과적인가"를 실험적으로 보여준 점은 가치 있음.

## 3. 솔직한 정리: LLM이 실제로 하는 것 / 못 하는 것

### LLM이 실제로 잘하는 것

| 역할 | 구체적 행동 | 기존 대안 |
|------|------------|-----------|
| **코드 생성 오케스트레이터** | SQL/pandas 코드를 한 줄씩 생성, 기존 함수 호출 | 수동 코딩 |
| **데이터 해석자** | 시각화를 보고 "계절성 있음" 판단, 전처리 전략 선택 | 사람 분석가 |
| **역할극 협상** | 공급망 각 주체 역할을 맡아 자연어 협상 | 규칙 기반 ABS |
| **피처 선택** | 어떤 보조 데이터를 포함할지 메타데이터로 판단 | 수동 피처 엔지니어링 |
| **설명 생성** | 왜 이 모델을, 왜 이 결정을 했는지 자연어 설명 | (기존엔 없음) |

### LLM이 못 하는 것 (그래서 외부 도구에 위임)

| 행위 | 실제 수행자 |
|------|------------|
| 수학 계산 | pandas, numpy, 최적화 솔버 |
| 시계열 예측 모델 실행 | scikit-learn, Prophet, LSTM 등 기존 모델 |
| 이상치 탐지 | discord-based 알고리즘 (자동) |
| 하이퍼파라미터 탐색 | grid search / bayesian opt (코드) |
| 정확한 발주량 산출 | EOQ 공식, 최적화 솔버 |

### LLM을 쓸 때 주의할 것

| 문제 | 원인 | 완화 방법 |
|------|------|-----------|
| 평균 앵커링 | 학습 데이터 패턴 매칭 | 인지적 성찰 프롬프트 |
| 채찍효과 증폭 | 수요 과대 추정 | 상하류 정보 공유 |
| 프레이밍 효과 | 프롬프트 워딩에 민감 | 중립적 프롬프트 설계 |
| 환각 | 없는 데이터를 만들어냄 | 도구 호출로 실제 데이터만 사용 |

## 4. 결론: 진짜 혁신은 어디에 있는가

**혁신이 아닌 것:**
- "LLM으로 수요예측하니 6% 좋아졌다" → 데이터 선택을 LLM이 한 것일 뿐
- "LLM 에이전트가 공급망을 최적화한다" → 기존 솔버를 LLM이 호출한 것일 뿐

**진짜 혁신적인 접근:**

1. **분산 협상 에이전트** (Paper 2): 중앙 집중 최적화가 불가능한 상황(기업 간 정보 비대칭)에서 LLM이 자연어 협상으로 합의를 도출. 기존 규칙 기반 ABS로는 유연한 협상이 불가능했음.

2. **반복적 자기 질문 (SIR)**: CoT의 선형적 추론을 넘어, 구조화된 소크라테스식 대화로 다단계 공급망 의사결정의 질을 높임.

3. **멀티모달 시계열 진단** (TSci): 플롯 이미지를 LLM이 직접 읽어서 시각적 패턴으로 모델/전처리를 선택. 기존 AutoML은 통계 피처만 사용.

4. **편향 발견** (AIM-Bench): LLM 재고 에이전트의 체계적 편향을 밝혀냄. 실무 적용의 가드레일 설계에 필수적인 발견.
