# 사용 가능한 공개 데이터셋 목록

> 목적: LLM 기반 공장 이상치 원인 추론 시스템 개발을 위한 데이터셋 검토
> 핵심 기준: **원인까지 알 수 있는가** (이상치 레이블만으로는 LLM 추론 검증 불가)

---

## 사용 가능 기준

| 기준 | 설명 |
|------|------|
| 시계열 센서 데이터 | 시간 흐름에 따른 수치 변화가 있어야 함 |
| 원인 레이블 | "이상치 있음/없음"이 아니라 **왜** 이상치가 생겼는지가 있어야 함 |
| 검증 가능성 | LLM의 추론 결과가 맞는지 틀린지 판단할 ground truth가 있어야 함 |
| 무료 접근 | 신청 없이 바로 다운로드 가능해야 함 (토이 프로젝트 기준) |

## Tier 1 — 바로 사용 가능 (원인 검증 가능)

---

### 1. AI4I 2020 Predictive Maintenance Dataset

**한 줄 요약**: 공장 설비 고장 예측용 가상 데이터. 5가지 실패 유형의 원인 규칙이 수식으로 공개됨.

**데이터 구성**
- 10,000행, 14개 컬럼
- 센서: 공기 온도(K), 공정 온도(K), 회전 속도(RPM), 토크(Nm), 공구 마모(min)
- 레이블: 5가지 실패 유형 각각 별도 컬럼으로 존재

**실패 유형과 원인 (공개된 규칙)**
| 실패 유형 | 원인 규칙 |
|----------|-----------|
| HDF (열 방출 실패) | 공기온도-공정온도 차이 < 8.6K AND 회전속도 < 1380 RPM |
| PWF (전력 실패) | 전력(=토크×RPM×2π/60) < 3500W 또는 > 9000W |
| OSF (과부하 실패) | 공구마모 × 토크 초과 (제품 등급별 임계값 다름) |
| TWF (공구 마모 실패) | 공구 마모 200-240분 구간 내 확률적 발생 |
| RNF (랜덤 실패) | 0.1% 확률로 무조건 발생 |

**왜 사용 가능한가**
- 원인 규칙이 수식으로 완전히 공개 → LLM 추론 결과를 정확히 검증 가능
- 센서 수가 적고 의미가 직관적 → 프롬프트 설계가 쉬움
- Kaggle/UCI 무료 다운로드

**무엇이 필요한가**
- 특별한 전처리 없이 바로 사용 가능
- 시스템 프롬프트에 각 실패 유형의 물리적 의미 설명 추가 권장
- 이벤트 로그 없음 → 센서 수치만으로 추론해야 함

**알려진 한계 (개발하다 부딪힐 것들)**
- 가상 데이터라 실제 공장의 복잡한 인과관계와 다름
- 원인 규칙이 단순해서 LLM이 너무 쉽게 맞출 수 있음 (난이도 검증 한계)
- 시간 흐름 무시 (행 순서가 시계열 아님) → 시간적 패턴 추론 불가
- RNF는 원인 없음 → 이 유형은 추론 검증에서 제외해야 함

**다운로드**
- Kaggle: https://www.kaggle.com/datasets/stephanmatzka/predictive-maintenance-dataset-ai4i-2020
- UCI: https://archive.ics.uci.edu/dataset/601/ai4i+2020+predictive+maintenance+dataset

### 2. Tennessee Eastman Process (TEP)

**한 줄 요약**: 화학 공정 시뮬레이터 데이터. 21가지 고장 유형 중 14개는 물리적 원인이 문서화됨.

**데이터 구성**
- 52개 변수 (22개 연속 측정값, 19개 조성 측정값, 11개 조작 변수)
- 샘플링: 3분 간격
- 훈련: 25시간, 테스트: 48시간 (고장은 훈련 1시간 후, 테스트 8시간 후 주입)
- 21개 고장 클래스별 분리된 파일

**고장 유형과 원인**
| 고장 | 종류 | 원인 |
|------|------|------|
| F1 | 스텝 변화 | A/C 피드 비율 (B 조성 상수) |
| F2 | 스텝 변화 | 피드 내 B 조성 |
| F4 | 스텝 변화 | 반응기 냉각수 입구 온도 |
| F5 | 스텝 변화 | 응축기 냉각수 입구 온도 |
| F6 | 스텝 변화 | A 피드 유실 (밸브 완전 개방) |
| F7 | 스텝 변화 | C 헤더 압력 유실 |
| F14 | 고착 | 반응기 냉각수 밸브 고착 |
| F15-F21 | 미지 교란 | 원인 미공개 (원 논문에서도 불명) |

**왜 사용 가능한가**
- F1-F14: 원인이 특정 공정 변수나 밸브로 명확히 지정됨
- 다변수 인과 연쇄 추론 가능 ("B 농도 변화 → 반응기 온도 → 압축기 부하" 같은 흐름)
- 가장 현실적인 공정 데이터 → 실제 공장 데이터 적용 가능성 검증에 적합
- Kaggle 무료

**무엇이 필요한가**
- 52개 변수의 의미를 담은 변수명 매핑 테이블 필요 (원 논문 또는 README 참조)
- 각 고장 유형의 물리적 의미를 시스템 프롬프트에 제공해야 함
- 시계열 구조이므로 특정 시점 기준 슬라이싱 전처리 필요

**알려진 한계**
- F15-F21은 원인 불명 → 이 클래스는 추론 검증에서 제외
- 화학 공정 도메인 → 공장(기계/설비) 도메인과 다소 다름
- 변수 수가 많아 LLM 컨텍스트 관리 복잡
- 이벤트 로그 없음 (순수 센서 시계열)

**다운로드**
- Kaggle: https://www.kaggle.com/datasets/averkij/tennessee-eastman-process-simulation-dataset
- Harvard Dataverse (확장판): https://dataverse.harvard.edu/dataset.xhtml?persistentId=doi:10.7910/DVN/6C3JR1

### 3. UCI Hydraulic System Condition Monitoring

**한 줄 요약**: 유압 시스템 테스트 장비 데이터. 4개 부품이 독립적으로 다른 심각도를 가짐.

**데이터 구성**
- 2,205 사이클 × 60초
- 센서: 압력(bar), 유량(L/min), 온도(℃) — 각기 다른 샘플링 주기 (1Hz~100Hz)
- 4개 부품 상태 레이블 (독립적):
  - 냉각기: 3(고장 임박)/20(성능 저하)/100(정상)
  - 밸브: 100(최적)/90(경미한 지연)/80(심각한 지연)/73(고장 임박)
  - 내부 펌프 누수: 0(없음)/1(약함)/2(심각)
  - 유압 어큐뮬레이터: 130(최적)/115(약간 저하)/100(심각 저하)/90(고장 임박)

**왜 사용 가능한가**
- 4개 부품이 동시에 다른 상태 → LLM이 "어떤 부품이, 얼마나 심각하게" 두 가지를 함께 추론해야 함
- 물리적 해석 가능: 유량 감소 = 펌프 누수, 압력 변동 = 밸브 지연, 온도 상승 = 냉각기 저하
- UCI/Kaggle 무료

**무엇이 필요한가**
- 다른 샘플링 주기를 가진 센서들의 시간 동기화 전처리 필요
- 각 부품-센서 연관성을 시스템 프롬프트에 명시해야 함
- 사이클 단위 집계 (평균/최대/표준편차) 전처리 권장

**알려진 한계**
- 이벤트 로그 없음
- 공장 제조 라인보다는 유압 시스템 특화 → 도메인 전이 필요
- 레이블이 심각도 등급이라 "원인 이벤트" 개념이 아님
- 멀티 샘플링 주기 전처리가 번거로움

**다운로드**
- UCI: https://archive.ics.uci.edu/dataset/447/condition+monitoring+of+hydraulic+systems
- Zenodo: https://zenodo.org/records/1323611

### 4. CATS (Controlled Anomalies Time Series Dataset)

**한 줄 요약**: 시뮬레이션 시스템에 200개 이상치를 주입한 데이터. 유일하게 원인 채널과 파급 채널을 구분해서 레이블링.

**데이터 구성**
- 17개 변수: 4개 제어 명령, 3개 환경 자극, 10개 텔레메트리
- 200개 이상치, 각각 시작/종료 타임스탬프 명시
- 레이블: 이상치 발생 채널 + **최초 원인 채널** + 파급 영향을 받은 채널 목록

**왜 사용 가능한가**
- "원인 채널 vs. 영향 채널" 구분이 유일하게 레이블로 존재
- LLM이 상관관계를 인과관계로 착각하는지 테스트 가능
- "A와 B가 같이 이상하다 → LLM이 A를 원인으로 지목하는가, B를 지목하는가" 검증 가능
- Kaggle/HuggingFace 무료

**무엇이 필요한가**
- 도메인이 명확하지 않은 시뮬레이션 시스템 → 시스템 프롬프트에 변수 의미 직접 정의해야 함
- 이상치 종류가 다양해서 유형별 분류 전처리 필요

**알려진 한계**
- 실제 공장 도메인 아님 → 도메인 특화 추론 테스트 한계
- 변수 의미가 추상적 → "물리적으로 말이 되는 설명" 유도가 어려울 수 있음
- 공장 LLM 시스템에 직접 연결하기 어렵고, 추론 능력 자체 검증에만 적합

**다운로드**
- Kaggle: https://www.kaggle.com/datasets/patrickfleith/controlled-anomalies-time-series-dataset
- HuggingFace: https://huggingface.co/datasets/patrickfleith/controlled-anomalies-time-series-dataset

## Tier 2 — 조건부 사용 가능 (전처리 또는 도메인 지식 추가 필요)

| 데이터셋 | 사용 가능 조건 | 주요 한계 |
|----------|---------------|-----------|
| **SKAB** | 단순 검증용 | 고장 유형이 2가지뿐 (밸브 개/폐). 다양성 없음 |
| **NASA CMAPSS** | RUL 예측 + 추론 | 원인이 데이터셋 레벨 (HPC/Fan 중 하나)만 명시. 샘플별 원인 없음 |
| **CWRU Bearing** | FFT 전처리 후 | Raw 진동 신호 → 주파수 분석 후에야 LLM에게 줄 수 있음 |
| **SWaT** | 연구용 신청 후 | 접근 승인 필요 (이메일 신청). ICS 보안 도메인 |

## 개발 단계별 권장 순서

```
Step 1  AI4I 2020
        이유: 단순, 원인 규칙 명확, 전처리 최소
        검증 목표: LLM이 5개 실패 유형 중 올바른 원인을 추론하는가?

Step 2  Tennessee Eastman (F1-F14)
        이유: 다변수 인과 연쇄, 현실적 공정
        검증 목표: LLM이 52개 변수 중 원인 변수를 추적하는가?

Step 3  UCI Hydraulic + CATS
        이유: 멀티 부품 동시 추론 / 원인-파급 구분
        검증 목표: LLM이 복잡한 인과 구조에서도 통찰을 찾는가?
```

## 공통 부족한 것 (모든 데이터셋)

- **이벤트 로그 없음**: 모든 Tier 1 데이터셋은 순수 센서 데이터. 실제 목표인 "센서 이상 + 유지보수 로그 교차 분석"은 별도로 시뮬레이션해야 함
- **해결책 평가 데이터 없음**: "해결책 시행 후 결과" 데이터는 어느 공개 데이터셋에도 없음. 직접 시뮬레이션 설계 필요
- **도메인 특화 지식**: 각 데이터셋의 물리적 의미를 LLM에게 제공하는 시스템 프롬프트 설계가 결국 품질을 결정함
