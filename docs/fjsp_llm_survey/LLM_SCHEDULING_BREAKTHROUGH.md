> 조사일: 2026-03-05

## 0. 왜 숫자로 분류하면 안 되는가

LLM 스케줄링 논문의 비교 대상 문제:

```
흔한 베이스라인                    실제 의미
─────────────────────────────────────────────────
SPT, EDD 같은 1950년대 룰         누가 만들어도 이김
"GPT-4에게 그냥 물어보기"          아무 최적화 없음, 당연히 이김
10년 전 DRL 방법                  당시 SOTA였지만 지금은 아님
자기 방법에서 모듈만 뺀 ablation   "부품이 도움된다" 수준의 주장
```

따라서 "X% 개선"이라는 숫자 대신, **이 연구가 없었으면 세상에 존재하지 않았을 것이 무엇인가**로 분류한다.

---

## 성과 스타일 분류 체계

| 스타일 | 핵심 질문 | 베이스라인 신뢰도 |
|--------|----------|----------------|
| **A. 발견** | 인간이 몰랐던 알고리즘/구조가 생겼는가? | 높음 (외부 검증 가능) |
| **B. 인터페이스** | 누가 이것을 쓸 수 있는가가 바뀌었는가? | 중간 (태스크 설정 의존) |
| **C. 자동화** | 전문가의 설계 노동을 줄였는가? | 낮음 (베이스라인 불명확) |
| **D. 파이프라인 개선** | 기존 시스템의 부품 하나를 더 잘 만들었는가? | 낮음 (ablation 위주) |
| **E. 패러다임 탐색** | 이 방향이 원리적으로 작동하는가를 탐색 | 중간 (이정표로 읽어야) |

---

## 스타일 A: 발견 — 없던 지식이 생겼다

### A-1. FunSearch (Google DeepMind, Nature 2023.12)

| 항목 | 내용 |
|------|------|
| **논문** | Mathematical discoveries from program search with large language models |
| **게재** | Nature 620 (2023) — 수학자들이 독립 검증 |
| **문제** | 캡셋 문제 + 온라인 빈 패킹 |

**성과:**
- 캡셋 문제: 20년간 깨지지 않던 하한 기록 돌파
- 빈 패킹: First Fit, Best Fit 등 전통 알고리즘을 다양한 분포에서 능가
- 단위 배정(Unit Commitment): 유전 알고리즘 대비 낮은 운영비용 + 빠른 실행

**왜 A등급인가:**
단순 성능 비교가 아니라 수학자들이 "이 구조는 우리가 생각하지 못했다"고 인정한 결과다. 출력물이 읽을 수 있는 코드이므로 인간이 이해하고 배울 수 있다.

**베이스라인 신뢰도:** 높음. Nature 수학 전문가 심사 + 외부 독립 검증.

---

### A-2. VRPAgent (arXiv, 2025.10)

| 항목 | 내용 |
|------|------|
| **논문** | VRPAgent: LLM-Driven Discovery of Heuristic Operators for Vehicle Routing Problems |
| **문제** | CVRP, VRPTW, PCVRP (2,000 고객 규모) |

**성과:**
- SOTA인 SISRs 대비 gap -0.30% (즉 SISRs보다 나음)
- 인간 전문가 설계 연산자를 자동 발견해 능가
- 단일 CPU 코어 사용

**왜 A등급인가:**
SISRs는 산업 현장에서 실제로 사용되는 검증된 방법이다. 약한 베이스라인이 아니다.

**베이스라인 신뢰도:** 높음. 논문이 직접 SOTA와 비교.

---

### A-3. EDDC/MDDC 발견 (arXiv 2510.24013, 2025.10)

| 항목 | 내용 |
|------|------|
| **논문** | Discovering Heuristics with LLMs for Mixed-Integer Programs: Single-Machine Scheduling |
| **문제** | 단일 기계 총 납기지연 최소화 (SMTT) |

**성과:**
- LLM이 EDD, MDD에서 영감 받은 새 휴리스틱 EDDC, MDDC 자동 발견
- 500개 작업 규모에서 기존 고전 휴리스틱 능가
- MDDC는 대규모 인스턴스에서 정확해(exact method)와 경쟁

**FunSearch와의 차이:**
FunSearch는 수학적으로 완전히 새로운 구조를 발견했고, 이 논문은 기존 아이디어에서 영감받은 "변형 발견"에 가깝다. A등급이지만 FunSearch보다는 약한 A다.

---

## 스타일 B: 인터페이스 — 누가 쓸 수 있는가가 바뀌었다

### B-1. MeetMate / "I Want It That Way" (ACM TiiS, 2023~2024)

| 항목 | 내용 |
|------|------|
| **논문** | "I Want It That Way": Enabling Interactive Decision Support Using LLMs and Constraint Programming |
| **게재** | ACM Transactions on Interactive Intelligent Systems |
| **문제** | 회의 스케줄링 + 인간의 비정형 선호 처리 |

**이 논문이 특별한 이유:**
유저가 처음에 기대했던 **"숫자로 측정되지 않는 인간 선호를 제약으로 변환"**을 직접 다룬다.

```
사용자 발화:
"월요일 오전은 집중 업무 시간이라 회의 잡지 마"
"팀장님과 1:1은 가급적 오후에"
"점심 전후 30분은 비워줘"

LLM이 하는 일:
이 자연어 선호들을 CP(제약 프로그래밍) 제약식으로 변환
→ 실제 스케줄러가 이 제약을 지키며 최적화
```

**연구 방법론이 탄탄한 이유:**
- 일기 연구(n=64): 실제 직장인들이 어떤 스케줄 선호를 갖는지 먼저 조사
- 정량 평가: LLM이 생성한 제약식을 전문가 구현과 비교 → 높은 precision/recall
- 사용자 연구(n=10): 실제 프로토타입 사용 경험 평가

**베이스라인 신뢰도:** 높음. 성능 비교가 아닌 사용자 연구 기반.

---

### B-2. MIT LLMFP (MIT, 2025.04)

| 항목 | 내용 |
|------|------|
| **논문** | Planning Anything with Rigor: General-purpose Zero-shot Planning with LLM-based Formalized Programming |
| **적용** | 창고 로봇 경로, 항공사 승무원, 공장 기계 스케줄링 등 9개 문제 |

**성과:**
- 성공률 85% vs 기존 최고 39%

**작동 방식:**
```
자연어 → LLM이 수학 모델로 변환 → OR-Tools 등 전문 솔버가 품
```
LLM은 직접 최적화하지 않는다. **"수학적 모델링" 과정을 자동화**한 것이다.

**MeetMate와의 차이:**
MeetMate는 "인간 선호의 비정형성" 처리에 집중,
MIT LLMFP는 "다양한 최적화 문제 유형의 범용 처리"에 집중.
둘 다 B스타일이지만 지향점이 다르다.

---

### B-3. OR-LLM-Agent (arXiv 2503.10009, 2025.03)

| 항목 | 내용 |
|------|------|
| **논문** | OR-LLM-Agent: Automating Modeling and Solving of Operations Research Optimization Problem |
| **특징** | 자연어 OR 문제 → Gurobi 코드 자동 생성 |

**성과:** 테스트 케이스 100% 통과, 솔루션 정확도 85%

**의의:** MIT LLMFP의 방향과 같으나, 출력물이 실행 가능한 Gurobi 코드라 엔지니어가 검토·수정 가능하다는 실용적 장점.

---

## 스타일 C: 자동화 — 전문가의 설계 노동을 줄였다

### 해당 논문들: FJSP 서베이 8편 (SeEvo, LLM4DRD, HFLLMDRL 등)

**무엇을 자동화했는가:**
- 디스패칭 룰 Python 코드 자동 생성·진화 (SeEvo, LLM4DRD, HRC-LLM)
- DRL 보상 함수 자동 설계 (HFLLMDRL)

**왜 베이스라인 신뢰도가 낮은가:**

```
논문이 이겼다고 주장하는 대상:
  SPT, EDD (1950년대 룰)         → 당연히 이김
  수작업 보상 함수               → 어떤 수준의 수작업인지 불명확
  기본 GP, GEP                   → 약한 기준선

논문이 비교하지 않은 대상:
  현장 엔지니어가 며칠 고민한 룰
  Gurobi/OR-Tools 직접 적용
  최신 Neural CO 방법들
```

**실제 의미:**
"LLM이 자동으로 설계한 것이 초보적인 룰보다 낫다"는 것은 증명했다.
"LLM이 전문가보다 낫다"는 아직 증명되지 않았다.

---

## 스타일 D: 파이프라인 개선 — 부품을 더 잘 만들었다

### 해당 논문들: ReflecSched, LLM4EO, NS4S, HeurAgenix

**공통 구조:**
기존 알고리즘 프레임워크에서 특정 모듈을 LLM으로 교체 → 성능 향상 주장

**NS4S (IJCAI 2025) 예시:**
- 로컬 서치의 이웃 평가 가중치 조정을 LLM으로 교체
- SchedAgent 기준 10초 내 84%, 30초 내 90% 랭킹

**HeurAgenix (Microsoft, 2025.06):**
- 2단계: 진화(LLM이 휴리스틱 개선) + 선택(LLM이 상황에 맞는 휴리스틱 고름)
- 전문 솔버(Gurobi, OR-Tools)와 동등하거나 초과 주장
- 단, "동등하거나 초과"하는 문제 인스턴스의 범위가 선택적일 수 있음

**베이스라인 신뢰도 이슈:**
대부분 ablation (모듈 빼고 비교) 위주. "LLM 모듈이 없는 것보다 낫다"는 증명이지, "기존 최고 방법보다 낫다"는 증명이 아닌 경우가 많다.

---

## 스타일 E: 패러다임 탐색 — 이 방향이 가능한가를 실험

### E-1. Starjob (2025.03)

**질문:** LLM이 JSSP를 end-to-end로 직접 풀 수 있는가?
**답:** 가능하지만 최적 대비 gap ~22%. 아직 멀다.

### E-2. NeurIPS 2025 End-to-end CO Solver (2025.09)

**질문:** 파인튜닝하면 gap이 얼마까지 줄어드나?
**답:** SFT + FOARL로 1.03~8.20%까지 가능.

**이 수치를 어떻게 읽어야 하나:**
Gurobi가 이 문제들에서 대부분 0~1% gap을 달성한다는 점을 감안하면, 8% gap은 아직 전문 솔버와 경쟁 수준이 아니다. 그러나 "LLM이 중간 단계 없이 직접 NP-hard 문제를 푼다"는 개념을 처음으로 실용적인 수준까지 끌어올렸다는 의미는 있다.

## 비공식 관행 추출: 가장 큰 미개척 영역

> 질문: 기존에 스케줄링하던 관행의 결과를 LLM에게 계속 보여주면서,
> 비공식 관행과 암묵적 규칙을 도출해내는 방식을 한 연구가 있는가?

### 결론: 이 방식 자체를 제조 스케줄링에 적용한 연구는 없다

유저가 상상한 흐름:
```
과거 스케줄링 로그 (수천 건)
  → LLM이 지속적으로 관찰
  → "이 공장은 월요일 오전에 납기 긴박한 작업을 우선한다"
  → "특정 작업자 조합이 배정되면 처리 속도가 빠르다"
  → 이런 비공식 관행을 규칙으로 명문화
```

이 전체 흐름을 제조 스케줄링에 적용한 논문은 존재하지 않는다.

---

### 조각조각 비슷한 방향의 연구

**RuAG (Microsoft, ICLR 2025) — 가장 구조적으로 유사**

```
대규모 과거 데이터
  ↓ MCTS로 조합 탐색
LLM이 1차 논리 규칙 자동 추출
  예: "IF 기계 부하 > 0.8 AND 납기 < 2일 THEN 우선순위 최상"
  ↓
이 규칙을 자연어로 변환 → LLM 프롬프트에 주입
```

F1 score 92.59% 달성. 단, 스케줄링 전용이 아닌 범용 프레임워크다.
제조 로그에 RuAG를 직접 적용한 연구는 없다.

**SMILO (arXiv 2511.02364, 2025.11) — 방향이 반대**

```
유저가 원한 방향: 결과 관찰 → 규칙 역추론 (bottom-up, 데이터 주도)
SMILO의 방향:    전문가가 규칙을 줌 → LLM이 수식으로 변환 (top-down)
```

SMILO는 이미 알고 있는 규칙을 MILP 모델로 바꿔주는 것이지,
숨겨진 관행을 스스로 찾아내는 것이 아니다.

**IRL (Inverse Reinforcement Learning) — 개념적으로 가장 가까움**

```
전문가 행동 시퀀스 → "이 사람은 어떤 기준으로 결정했는가" 역추론
                  → 암묵적 보상 함수 추출
```

스케줄링 IRL 연구는 존재하지만, LLM을 이 과정에 결합한 연구가 없다.

---

### 왜 이 연구가 없을까

```
필요한 것 1: 실제 공장의 스케줄링 로그 (대부분 비공개)
             - 작업 배정 이력, 작업자 결정 이력, 예외 처리 사례

필요한 것 2: "왜 그 결정을 했는가"에 대한 레이블
             - 로그만 있으면 결과만 보임
             - 이유는 대부분 사람 머릿속에만 있음 (암묵지)
```

공개 벤치마크는 거의 2번이 없다. 결과만 있고 이유가 없다.
따라서 지도학습도, 강화학습 보상 설계도 어렵다.

---

### 현재 기술로 구현한다면

RuAG + IRL + MIT LLMFP를 결합하는 구조가 가장 현실적이다:

```
[오프라인: 규칙 추출 단계]
과거 스케줄 로그 수천 건
  ↓ RuAG 방식 (MCTS + LLM)
패턴 탐색 → 자연어 규칙 추출
  예: "긴급 주문 발생 시 기계 Y를 기계 X보다 선호"
      "특정 작업자 조합 시 처리시간 단축 경향"

[온라인: 의사결정 단계]
새 스케줄링 상황 발생
  ↓ 추출된 규칙을 제약으로 변환 (MeetMate/MIT LLMFP 방식)
LLM + CP 솔버가 규칙을 지키며 최적화
```

이 파이프라인을 제조 스케줄링에 통째로 구현한 논문이 없다.
**현재 기준으로 명확한 연구 공백이다.**

---

## 조사를 통해 드러난 공백

### 공백 1: 제조 스케줄링 특화 "발견" 연구가 거의 없다

FunSearch는 수학/빈패킹, VRPAgent는 VRP, EDDC/MDDC는 단일 기계 문제다.
**FJSP/DFJSP 특화 "발견"은 아직 없다.** FJSP 논문들은 전부 C/D 스타일이다.

### 공백 2: 인간 선호의 비정형성 처리가 제조에는 미적용

MeetMate는 회의 스케줄링에서 성공적으로 구현됐지만,
**공장 환경(작업자 선호, 안전 조건, 비공식 관행)에 적용한 연구가 없다.**

### 공백 3: 강한 베이스라인과의 비교가 없다

Gurobi, OR-Tools와 직접 비교한 LLM 스케줄링 논문이 거의 없다.
대부분 "LLM 없는 버전"이나 "고전 휴리스틱"과 비교한다.
**진짜 최강 도구와 비교했을 때 LLM이 어떤 상황에서 이기고 지는지 모른다.**

### 공백 4: 재현성과 평가 공정성 문제

LLM 출력의 확률적 특성 때문에 동일 실험 재현이 어렵다.
API를 통해 GPT-4를 쓰는 경우, 모델 업데이트로 결과가 달라질 수 있다.
이를 체계적으로 다룬 논문이 없다.
