# LLM 기반 공장 이상치 원인 추론 — 토이 프로젝트

> **검증 질문**: LLM은 공장 센서 데이터만 보고 고장 원인을 추론할 수 있는가?

---

## Step 1: AI4I 2020 데이터셋 실험

### 데이터셋
- AI4I 2020 Predictive Maintenance Dataset (Kaggle/UCI)
- 10,000행, 5개 센서, 5가지 고장 유형
- 고장 판별 규칙이 수식으로 완전 공개된 가상 데이터

### 구현한 것

```
step1_failure_reasoning/app.py   Streamlit 단일 파일 앱
data/ai4i2020.csv                데이터셋 (수동 다운로드)
```

**앱 구조 (3탭)**

| 탭 | 내용 |
|----|------|
| 1단계: 메타데이터 분석 | 통계 요약을 LLM에게 전달 → 어떤 변수가 고장 구분에 유효한지 분석 |
| 2단계: ML 학습 + 해석 | 의사결정나무로 분류 규칙 발견 → LLM이 규칙을 물리적으로 해석 |
| 3단계: 샘플 설명 | ML이 새 샘플 분류 → LLM이 원인과 조치를 운영자 언어로 설명 |

**파생 변수 (물리 법칙 기반)**
- `온도차 = 공정온도 - 공기온도` → 열 방출 효율 지표
- `전력 = 토크 × RPM × 2π/60` → 실제 소비 전력
- `마모_토크 = 공구마모 × 토크` → 기계적 누적 부하

### 실험 과정에서 바뀐 접근

| 시도 | 문제 | 결론 |
|------|------|------|
| Zero-shot: 센서 수치 5개만 LLM에 전달 | 통계적 기준이 없어 물리 상식으로 때려맞춤 | LLM이 수치 맥락 없이 추론 불가 |
| In-context: 전체 고장 데이터(330행) 프롬프트에 삽입 | LLM을 분류 엔진으로 쓰는 구조 — ML보다 느리고 비쌈 | 잘못된 LLM 활용 방식 |
| ML 분류 + LLM 해석 | ML이 패턴 발견, LLM이 물리적 의미 해석 | 역할 분리는 맞지만 데이터가 너무 단순 |

### 핵심 발견

**이 데이터셋에서 LLM의 역할은 제한적이다.**

AI4I 2020은 ML 분류 벤치마크용으로 설계된 데이터셋이다. 센서 수치 → 고장 유형 분류는 의사결정나무만으로 충분하며, LLM은 ML이 찾은 답을 말로 설명하는 역할밖에 없다.

LLM이 진짜 힘을 발휘하는 조건:
- 정비 일지, 작업자 보고, 설비 매뉴얼 등 **비정형 텍스트**가 함께 있을 때
- 센서 이상과 운영 이벤트를 **교차 분석**해야 할 때
- 단순 분류가 아닌 **인과 연쇄 추론**이 필요할 때

이 데이터셋은 그 어느 것도 없다.
